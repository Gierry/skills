# 微信公众号文案转写 (步骤1)

将原始文案转写为结构化的Markdown格式,为后续配图和HTML转换做准备。

## 使用场景

- 用户提供原始文案,需要改写成公众号风格
- 网络文本需要转化为微信生态阅读习惯
- 需要生成爆款标题和清晰结构的文章

## 工作流程

### 进度检查清单

```
文案转写进度:
- [ ] Step 1: 接收并分析原始文案
- [ ] Step 2: 生成爆款标题组
- [ ] Step 3: 创作结构化正文
- [ ] Step 4: 保存Markdown文件
- [ ] Step 5: 输出完成报告
```

### Step 1: 接收并分析原始文案

**输入方式**:
1. 用户直接粘贴文案内容
2. 用户提供文件路径

**分析内容**:
- 识别文案类型(教程/评测/观点/故事)
- 提取核心观点和关键信息
- 判断目标受众和情绪基调
- 确定适合的写作风格

### Step 2: 生成爆款标题组

**标题类型** (生成3-5个不同风格):

| 类型 | 特点 | 示例 |
|------|------|------|
| 悬念类 | 引发好奇 | "99%的人都不知道的..." |
| 利益点类 | 直接价值 | "3个方法让你..." |
| 情绪共鸣类 | 戳中痛点 | "终于有人说出了..." |
| 数字类 | 具体量化 | "5分钟学会..." |
| 对比类 | 制造反差 | "从XX到XX,我..." |

**标题原则**:
- 长度控制在15-25字
- 包含核心关键词
- 避免标题党和夸张
- 符合微信生态规范

### Step 3: 创作结构化正文

**使用Markdown标记**:

| 内容类型 | 标记方式 | 说明 |
|---------|---------|------|
| 章节标题 | `## 标题` | 主要章节 |
| 小节标题 | `### 标题` | 次级章节 |
| 普通段落 | 直接文字 | 每段不超过3行 |
| 重点强调 | `**文字**` | 核心内容加粗 |
| 列表项 | `- 内容` | 并列要点 |
| 引用/金句 | `> 内容` | 头部金句或引用 |
| 代码/命令 | ` ```bash ` | 真正的代码 |
| 重点提醒 | `> ⚠️ **重点**：内容` | 特别提醒 |

**⚠️ 代码块使用规则**:

✅ **正确用法** - 只有真正的代码/命令才用代码块:
```bash
npm install -g package-name
```

❌ **错误用法** - 不要用代码块包裹非代码内容:
```
这是重点提醒 ← 错误!
```

✅ **正确替代** - 重点提醒用引用块:
> ⚠️ **重点**：这是重点提醒内容

**文章结构模板**:

```markdown
# 标题选项

1. [标题选项1] (推荐)
2. [标题选项2]
3. [标题选项3]

---

> [头部金句 - 一句能戳中人心的核心观点]

## 01 [第一章节标题]

[开篇段落,引入话题]

### [小节标题]

[具体内容,每段不超过3行]

**核心观点**用加粗强调。

- 要点1
- 要点2
- 要点3

> ⚠️ **重点**：特别需要注意的内容

## 02 [第二章节标题]

[继续展开...]

---

## 互动尾部

[一句话总结]

**留言区聊聊**：[引发读者留言的问题]
```

**写作准则**:
- **情绪调动**: 拒绝枯燥,采用幽默风趣或专业严谨的语调
- **内容提炼**: 去除赘述,保留核心干货
- **段落呼吸**: 每段不超过3行,保持阅读节奏
- **重点突出**: 核心结论用 `**加粗**` 强调
- **逻辑清晰**: 章节之间有明确的递进关系

### Step 4: 保存Markdown文件

**文件命名规则**:

1. 从标题提取关键词(2-4个词)
2. 转换为kebab-case格式
3. 添加日期前缀

**示例**:
- 标题: "Claude Code完全指南" → `2024-01-15-claude-code-guide.md`
- 标题: "职场新人成长指南" → `2024-01-15-workplace-growth-guide.md`

**保存位置**:

```
wechat-articles/
└── YYYY-MM-DD/
    └── {slug}.md
```

**冲突处理**:
- 如果文件已存在,备份为 `{slug}-backup-YYYYMMDD-HHMMSS.md`
- 然后保存新文件

**文件内容格式**:

```markdown
---
title: [最终选定的标题]
date: YYYY-MM-DD
slug: [文件名slug]
status: draft
---

[完整的Markdown内容]
```


## 注意事项

### ✅ 必须做到

1. **保持信息完整**: 原文的关键信息点必须保留
2. **语言口语化**: 用亲切、易懂的方式表达
3. **结构清晰**: 章节层级分明,逻辑递进
4. **标记规范**: 严格按照Markdown语法
5. **代码块限制**: 只用于真正的代码和命令

### ❌ 禁止行为

1. **不要输出开场白**: 如"当然可以!"、"好的,我来帮你..."
2. **不要过度使用Markdown**: 保持简洁,不要滥用格式
3. **不要改变核心观点**: 可以换表达方式,但不能改变原意
4. **不要添加无关内容**: 只改写用户提供的内容
5. **不要使用复杂嵌套**: 避免多层嵌套的列表或引用
